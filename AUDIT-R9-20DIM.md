## Verdict

**110 findings survived verification.** Feature coverage is genuinely deep — the defining result of R9 is that **many of the most severe findings are controls that exist in code, are documented as enforced, and never actually execute**:

- SoD gates reading fields Mongoose silently drops (`payments[].receivedById`)
- state machines registered as `pre("save")` hooks on paths that write via `findOneAndUpdate`
- URL-regex authZ walls defeated by Express's case-insensitive router
- an allergy gate fed a field name the schema never declares
- a maker-checker on credit notes that **both** GST exporters ignore

The second-largest severity source is **sibling drift**: prior rounds (R7*/R8) repeatedly fixed one path and not its twin. Net effect — the compliance register **overstates coverage**, the riskiest posture for an NABH/NABL/GST/NDPS product sold to 100+ sites.

## Cross-cutting themes

1. **Sibling drift is the dominant defect generator.** Prior fixes landed on one path, not its twin: create-vs-update strips (patient, OPD, MLC, discharge summary, nursing care plan, LabReport), GSTR-1 vs GSTR-3B (`R8-FIX(#2)`), `billNumber` vs `generateOPDAdmissionNumber` (the known `#38` sibling), FallRisk `key` remount vs the other nursing pages (the `#47` fix). **Remediate twin-scoped, not site-scoped.**
2. **Controls that cannot execute.** Guards attached as `pre("save")` while writers use `findOneAndUpdate`/`updateOne`; fields read that the schema never declares (`receivedById`, `drugName`, `tpaApprovedLimitPerUnit`); `limit(0)` meaning *unlimited*; a client-side-only transaction "capability probe". The register reads green over dead code.
3. **Client-supplied identity and flags.** `verifiedBy`, `enteredBy`, `finalizedByName`, `signedBy`, `approvedAmount`, `retainUntil` come from the request body — collapsing NABL authorising-signatory, NABH co-sign and sentinel idempotency into values the constrained party controls.
4. **Standalone-Mongo diverges from replica-set.** The per-hospital Docker model runs the non-transaction branches: MLC `createdById` never persists (SoD dead), admission rollback frees **another patient's** bed, pharmacy advance-apply 500s every call.
5. **Void/reversal semantics are unmodelled** — Day Book, AR aging and todayRevenue each double-count or mis-sign the same voided payment.

## CRITICAL (11)

### R9-080 — Express case-insensitive routing defeats the case-sensitive URL-regex authZ walls (Doctor/Nurse money-write block + discharged-record seal)

**AuthN / AuthZ / RBAC / tokens / rate limiting** · security · CONFIRMED · confidence: high

**Location:** `Backend/middleware/auth.js` : 429-468 (blockNonClinicalForDoctorNurse; regexes 430-448, match at 455-457) and 691-812 (enforceActivePatientForClinicalWrites; rule match at 810-812)

**Mechanism:** Both walls are deny-on-match: they lowercase nothing and test hand-written case-sensitive regexes (e.g. `/\/billing\/[^?]*\/payment(\/|$)/`, `/\/cashier-sessions(\/|$)/`, `/\/doctor-notes(\/|$|\?)/`) against `req.originalUrl.split("?")[0]`. Express 5 (package.json:20) is mounted in Backend/index.js with no `app.set("case sensitive routing", true)` — grep for `caseSensitive`/`case sensitive` across Backend returns zero hits — so path-to-regexp matches routes with `sensitive:false` and `/api/BILLING/<id>/PAYMENT` dispatches to the same handler as `/api/billing/<id>/payment`. The regexes do not match the upper-cased URL, so the middleware calls `next()` and the wall is skipped. The route-level gate behind the wall does not compensate: `POST /:billId/payment` (routes/Billing/billingRoutes.js:168), `POST /advance` (:240), `POST /advance/:advanceId/apply` (:245), `POST /uhid/:UHID/collect-all` (:112) and `POST /cashier-sessions/open` (routes/Billing/cashierSessionRoutes.js:10) are all gated on `requireAction("billing.write")`, and `billing.write` explicitly includes Doctor and Nurse (config/permissions.js:251). The comment at permissions.js:243-250 states this is SAFE *because* blockNon…

**Failure scenario:** A logged-in Doctor (or Nurse) sends `POST /api/billing/<billId>/PAYMENT` with `{amount: 50000, paymentMode: "CASH"}` and their normal Bearer token. authenticate passes; blockNonClinicalForDoctorNurse sees url `/api/billing/<billId>/PAYMENT`, none of its 13 lower-case regexes match, so it calls next(); billingRoutes matches the route case-insensitively; requireAction("billing.write") allows Doctor; recordPayment writes a ₹50,000 CASH receipt stamped `receivedByRole: "Doctor"`, mutates the bill balance and the cashier day-book — a separation-of-duties control the codebase documents as enforced. The same trick opens `/api/CASHIER-SESSIONS/open` (Doctor opens a cash drawer) and `/api/ADVANCE`-style paths. Identically, `POST /api/DOCTOR-NOTES` with `admissionId` of a **Discharged** admission skips enforceActivePatientForClinicalWrites entirely (rule lookup at auth.js:810 finds no rule for the upper-cased url), silently writing into a NABH MOI.1-sealed record without the audited X-Late-Entry reason path (auth.js:889-910).

**Fix:** Normalize once before matching: `const url = req.originalUrl.split("?")[0].toLowerCase();` in blockNonClinicalForDoctorNurse (auth.js:455), enforceActivePatientForClinicalWrites (auth.js:807) and the resolvers that regex `req.path`, and/or set `app.set("case sensitive routing", true)` in Backend/index.js so the router itself refuses the upper-cased path. Belt-and-braces: add a role check inside recordPayment/openSession, or split a `billing.payment` action that excludes Doctor/Nurse rather than relying on a URL wall.

### R9-024 — PaymentSchema has no receivedById/receivedByRole/voidedById — Mongoose strict mode silently drops them, killing the refund segregation-of-duties gate and per-cashier shift reconciliation

**Billing core** · security · CONFIRMED · confidence: high

**Location:** `Backend/models/PatientBillModel/PatientBillModel.js` : 170-211 (PaymentSchema); consumers: services/Billing/billingService.js:1747-1748, 1887-1890, 1988-1996, 2103-2106; controllers/Billing/cashierSessionController.js:121-124, 160-172; index.js:582-602

**Mechanism:** PaymentSchema declares only receiptNumber, amount, paymentMode, transactionId, paidAt, receivedBy, remarks, voidedAt/voidedBy/voidedByRole/voidReason and the tds* fields. It does NOT declare `receivedById`, `receivedByRole`, or `voidedById`. Every writer pushes them anyway (recordPayment 1747, bulkCollect 1164, voidPayment 1887-1890, recordRefund 2103-2106) and Mongoose's default strict mode discards unknown subdocument paths on save — so `payments[].receivedById` is ALWAYS absent on disk. Two controls read it back: (a) recordRefund's SoD probe at billingService.js:1988-1996 selects `payments.receivedById` and refuses SAME_ACTOR only if it matches refundedById — with the field always undefined, `p.receivedById && ...` is always false, so the check never fires; (b) cashierSessionController.js:121-124/160-163 filters bills by `"payments.receivedById": session.cashierId`, which matches zero documents. Backend/scripts/backfillReceivedById.js:21 states its own precondition — "Run only after R7bb-FIX-B-1 has shipped (i.e. PaymentSchema has the field)" — which never landed; the script itself would also no-op because its `p.receivedById = candidate._id` assignment is dropped at save too.…

**Failure scenario:** Cashier A (role Accountant, has billing.refund) records a ₹40,000 CASH payment on bill BILL-26-14 — the receivedById she is authenticated as is dropped at save. Cashier A then POSTs /api/billing/BILL_ID/refund {amount: 40000, reason: "patient overpaid", mode: "CASH"} with no approverOverride. The SoD probe reads payments[].receivedById as undefined for every row, `sameActor` is false, and the refund proceeds — the same person collects and returns ₹40,000 cash with no second signature, which is exactly the control D3-CRIT-1 claims to enforce. Separately, when she closes her shift, `PatientBill.countDocuments({"payments.receivedById": cashierId, ...})` returns 0 and the close-window `find()` returns no bills, so cashCollected computes to 0 (plus advance deposits only) and the drawer variance report shows the entire shift's ₹40,000 of bill collections as unaccounted/invisible.

**Fix:** Add to PaymentSchema: `receivedById: { type: mongoose.Schema.Types.ObjectId, ref: "User", default: null }`, `receivedByRole: { type: String, trim: true, default: null }`, `voidedById: { type: mongoose.Schema.Types.ObjectId, ref: "User", default: null }` (mirroring PatientAdvanceModel.js:71-72). Add an index on `payments.receivedById` for the shift queries. Then run scripts/backfillReceivedById.js --dry-run. Until backfilled, the refund SoD probe should fail-closed (refuse when receivedById is missing on positive rows) rather than fail-open.

Consultant","finalizedAt":"2026-07-17T10:00:00Z","dischargeType":"Death"}. ### R9-040 — applyAdvanceToSale: per-sale idempotency skip credits the bill without debiting the advance — free money on any 2nd apply

**Mechanism:** The FIFO bleed loop skips an advance when `(adv.appliedTo || []).some(r => String(r.billId) === String(sale._id))` is true, and compensates by doing `need -= slice; continue;` — i.e. it treats the slice as already spent.
